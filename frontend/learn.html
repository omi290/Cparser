Compiler Design
All 7 Phases Explained

A compiler translates high-level source code into machine code through a structured pipeline of seven phases. Each phase performs a specific transformation, passing its output to the next stage.

Source → Lexical → Syntax → Semantic → IR Gen → Optimize → Code Gen → Exe
1

Lexical Analysis (Scanner / Tokenizer)

Breaking source code into meaningful tokens

The lexical analyzer reads the source code character by character and groups them into lexemes, producing a stream of tokens. It removes whitespace and comments, identifies keywords, identifiers, literals, operators, and separators.

int x = 42; → KEYWORD:int ID:x OP:= NUM:42 SEP:;

🔑 Key Concepts

Tokens, Patterns & Lexemes

A token is a category (KEYWORD, IDENTIFIER). A pattern is the rule that describes it (regex). A lexeme is the actual matched string ("int", "x").

Regular Expressions

Patterns for tokens are specified using regular expressions. E.g., [a-zA-Z_][a-zA-Z0-9_]* for identifiers, [0-9]+ for integers.

Finite Automata (DFA/NFA)

Regular expressions are converted to NFA (Thompson's construction), then to DFA (subset construction) for efficient scanning. The DFA drives the actual tokenizer.

Symbol Table Entries

When identifiers are recognized, they are entered into the symbol table — a data structure that stores variable names, types, scopes, and memory locations used throughout compilation.

Error Recovery

When invalid characters are encountered, the lexer uses strategies like panic mode (skip until valid token), deleting extraneous chars, or inserting missing ones.

Tools: Lex / Flex

Lex and Flex are popular lexer generators. You write token patterns, and they auto-generate a C scanner using DFA internals.

2

Syntax Analysis (Parser)

Building a structured tree from the token stream

The syntax analyzer (parser) takes the token stream from the lexer and verifies that it conforms to the grammar rules of the language. It produces a parse tree or Abstract Syntax Tree (AST) representing the hierarchical structure.

x = a + b * c; → Parse Tree with operator precedence: (x = (a + (b * c)))

🔑 Key Concepts

Context-Free Grammars (CFG)

Languages are defined by production rules: E → E + T | T, T → T * F | F, F → (E) | id. CFGs can express nested and recursive structures.

Parse Trees & AST

A parse tree shows every grammar derivation step. An AST is a simplified version that omits redundant nodes (parentheses, semicolons) keeping only semantics.

Top-Down Parsing (LL)

Recursive Descent and LL(1) parsers start from the root and predict productions. They are simple but can't handle left-recursive grammars directly.

Bottom-Up Parsing (LR)

SLR, CLR, and LALR parsers build the tree from leaves up using shift-reduce actions. LALR (used by Yacc/Bison) handles more grammars than LL.

Ambiguity & Precedence

Ambiguous grammars have multiple parse trees for the same input. Resolved via operator precedence, associativity rules, and grammar rewriting.

Tools: Yacc / Bison

Yacc (Yet Another Compiler Compiler) and Bison generate LALR(1) parsers from grammar specifications written in BNF notation.

3

Semantic Analysis (Type Checker)

Ensuring meaning and consistency beyond syntax

The semantic analyzer checks the AST for meaning-related errors that syntax analysis cannot detect. It verifies type compatibility, scope rules, and declaration before use — ensuring the program makes logical sense.

int x = "hello"; → ❌ Type Error: cannot assign string to int

🔑 Key Concepts

Type Checking

Static type checking verifies type compatibility at compile time. Ensures operations like int + float are handled with proper type coercion or flagged as errors.

Scope Resolution

Variables are resolved to their correct declaration based on lexical scoping rules. Inner scopes shadow outer declarations. The symbol table tracks nested scope chains.

Attribute Grammars

Synthesized attributes pass information up the tree; inherited attributes pass down. Used for type propagation, constant evaluation, and decoration of the AST.

Type Coercion & Casting

Implicit coercion (widening: int→float) happens automatically. Explicit casting requires programmer annotation: (float)x. Narrowing may lose data.

Declaration Checking

Ensures every variable is declared before use, functions have correct argument counts and types, and return statements match function signatures.

Annotated AST

The output is a decorated/annotated AST where each node carries type information, resolved references, and computed attributes for later phases.

4

Intermediate Code Generation (IR)

Translating AST to a machine-independent representation

The intermediate code generator converts the annotated AST into a machine-independent intermediate representation (IR). This allows the compiler to separate front-end (language-specific) from back-end (machine-specific) concerns.

a = b + c * d → t1 = c * d
t2 = b + t1
a = t2

🔑 Key Concepts

Three-Address Code (TAC)

Each instruction has at most three operands: x = y op z. Complex expressions are broken into simple temporaries. This is the most common IR form.

Quadruples & Triples

Quadruples: (op, arg1, arg2, result). Triples: (op, arg1, arg2) with implicit result numbering. Indirect triples add flexibility for reordering.

Syntax-Directed Translation

IR generation is driven by semantic actions attached to grammar rules. As the parser reduces a production, corresponding IR code is emitted using translation schemes.

DAGs (Directed Acyclic Graphs)

DAGs represent expressions by sharing common subexpressions. If a+b appears twice, the DAG reuses one node — enabling early optimization.

SSA Form

Static Single Assignment gives each variable exactly one definition. Phi functions merge values at control-flow joins. Modern compilers (LLVM, GCC) use SSA internally.

Control Flow Translation

Loops, if-else, and switch are translated using labels and goto statements. Boolean expressions use short-circuit evaluation with conditional jumps.

5

Code Optimization (Optimizer)

Improving code efficiency without changing behavior

The code optimizer transforms the intermediate code to produce faster, smaller, or more power-efficient output — all while preserving the program's observable behavior. Optimization can be machine-independent (on IR) or machine-dependent.

t1 = 4 * 2 → t1 = 8   (constant folding)
x = x + 0 → eliminated   (algebraic simplification)

🔑 Key Concepts

Constant Folding & Propagation

Folding: evaluate constant expressions at compile time (3*4→12). Propagation: replace variable uses with known constant values throughout.

Dead Code Elimination

Remove code that computes values never used, unreachable code after return, and branches with constant conditions. Reduces code size and execution time.

Loop Optimization

Loop-invariant code motion (move unchanging computations outside). Loop unrolling (reduce iteration overhead). Strength reduction (replace multiplication with addition).

Common Subexpression Elimination

If a+b is computed multiple times with unchanged values, compute once and reuse. Applied locally (within blocks) or globally (across blocks via data-flow analysis).

Inline Expansion

Replace function calls with the function body directly. Eliminates call overhead but increases code size. The compiler decides based on function size and call frequency.

Data-Flow Analysis

Reaching definitions, live variable analysis, and available expressions — iterative algorithms on the control-flow graph that enable advanced optimizations.

6

Code Generation (Back-End)

Translating IR into target machine instructions

The code generator maps the optimized IR to actual target machine instructions (x86, ARM, RISC-V). It handles register allocation, instruction selection, and instruction scheduling to produce efficient assembly or machine code.

t1 = a + b → MOV R1, a
ADD R1, b
MOV t1, R1

🔑 Key Concepts

Register Allocation

Map unlimited temporaries to limited physical registers. Uses graph coloring on the interference graph. Variables that are live simultaneously need different registers.

Instruction Selection

Choose the best machine instructions for each IR operation. Some CPUs have compound instructions (MADD) that combine multiple operations. Tree pattern matching automates this.

Instruction Scheduling

Reorder instructions to minimize pipeline stalls and exploit instruction-level parallelism. Must preserve data dependencies while maximizing throughput.

Peephole Optimization

A sliding window over generated code that replaces inefficient patterns with better ones: redundant loads, unnecessary jumps, strength reductions at the machine level.

Memory Management

Generates code for stack frame setup/teardown, parameter passing conventions, activation records, and static/heap allocation. Follows the target ABI.

Target Architectures

CISC (x86) vs RISC (ARM). Each has different register counts, addressing modes, and instruction formats. The code generator must adapt to each target's constraints.

7

Symbol Table Management (& Error Handling)

The cross-cutting backbone used by all phases

The symbol table is a central data structure used by all phases of the compiler. It stores information about identifiers — names, types, scopes, memory locations, and more. Error handling also spans all phases, detecting and reporting issues at each stage.

int count = 0; → Symbol Table Entry: { name:"count", type:"int", scope:"main", offset:0, line:1 }

🔑 Key Concepts

Hash Table Implementation

Most compilers use hash tables for O(1) average lookup. Keys are identifier names; values hold type, scope, offset, and other attributes. Collision handling via chaining.

Scope Management

Nested scopes form a scope stack. Entering a block pushes a new scope; leaving pops it. Lookup searches from innermost to outermost, implementing lexical scoping.

Type Information

Stores data types, array dimensions, struct/union layouts, function signatures, and return types. Enables type checking and memory layout calculations.

Storage Allocation

Records memory offsets, storage classes (auto, static, extern, register), and alignment requirements. Used by the code generator to emit correct memory accesses.

Error Detection & Recovery

Lexical errors (invalid chars), syntax errors (unexpected tokens), semantic errors (type mismatches), and logical errors. Recovery strategies: panic mode, phrase-level, error productions.

Error Reporting

Good compilers provide line numbers, column positions, highlighted code, and suggestions for fixes. Multiple errors reported in one pass enable faster debugging.
